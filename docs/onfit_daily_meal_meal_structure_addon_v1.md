# OnFit Daily Meal Page - Meal Structure Add-on Spec v1

## 문서 목적
이 문서는 **현재 이미 개발 반영된 daily-meal 페이지를 기준으로**,  
여기에 **식사 구조(하루 끼니 + 식사 시간) 섹션만 추가 구현**하기 위한 코덱스 CLI 전달용 MD 파일이다.

중요:
- 이 문서는 daily-meal 페이지 전체를 새로 만드는 문서가 아니다.
- 이미 반영된 목표 설정 영역, 상단 타이틀/설명, 상태 요약 카드, 검증/모달 로직은 유지한다.
- 이번 작업 범위는 **식사 구조 섹션 추가**에 한정한다.

---

## 1. 전제
현재 daily-meal 페이지에는 아래가 이미 구현되어 있다고 본다.

- 페이지 URL: `daily-meal`
- 카드 밖 최상단 타이틀
  - `Step3 어디까지, 언제까지 갈지 정해보세요`
- 카드 밖 최상단 설명
  - `현재 체중을 기준으로 목표 체중과 기간을 설정해보세요`
- 입력 카드 상단 상태 요약 카드
- 목표 설정 영역
  - 목표 체중 / 목표 기간
  - 유지일 때 목표 체중 입력칸 숨김
- 현재 체중 노출 규칙
- 목표 설정 검증 로직
- 재확인 모달 로직
- 페이지 하단 `다음` CTA

이번 작업은 위 내용을 건드리는 것이 아니라,  
**목표 설정 영역 아래에 식사 구조 섹션을 추가하는 작업**이다.

---

## 2. 작업 목표
daily-meal 페이지 안에서 사용자가 아래를 설정할 수 있게 만든다.

1. 하루 식사 개수
2. 각 식사 이름
3. 각 식사 시간
4. 식사 순서
5. 식사 추가 / 삭제

즉, 목표 설정 이후 바로 **식사 구조까지 같은 페이지에서 설정**하게 만드는 것이 목적이다.

---

## 3. 배치 위치
식사 구조 섹션은 **daily-meal 페이지의 기존 입력 카드 안**에 추가한다.

### 위치 규칙
- 목표 설정 영역 아래
- 기존 `다음` 버튼 위

### 섹션 구분 방식
- 목표 설정 영역과 식사 구조 섹션 사이에는
  - **충분한 상하 간격**
  - **얇은 구분선 1개**
를 넣어서 구분한다.

주의:
- 식사 구조를 별도 페이지로 분리하지 않는다
- 카드 밖으로 빼지 않는다
- 카드 안에 별도 서브카드를 여러 개 추가하지 않는다

---

## 4. 섹션 타이틀
### 타이틀
`하루에 얼마나 드실 수 있으세요?`

### 설명
- 없음

주의:
- 이 섹션은 별도 설명 문구를 넣지 않는다
- 타이틀만으로 역할이 전달되게 한다

---

## 5. 기본 식사 구조
처음 진입 시 기본 식사 3개를 세팅한다.

### 기본값
- `아침`
- `점심`
- `저녁`

### 식사 개수 규칙
- 최소 1개
- 최대 10개

주의:
- 사용자는 식사를 추가/삭제할 수 있어야 한다
- 단, 0개는 허용하지 않는다
- 10개를 초과해서 추가할 수 없게 한다

---

## 6. 식사 목록 구성
각 식사는 **이름 + 시간**으로 구성한다.

### 배치 방식
- **한 줄 배치**

### 비율
- 식사 이름: **7/10**
- 시간: **3/10**

예시:
- `아침                 08:00`
- `점심                 13:00`
- `저녁                 19:00`

주의:
- 이름과 시간을 두 줄로 쪼개지 않는다
- 식사 목록이 길어질 수 있으므로 세로 길이를 과도하게 늘리지 않는 배치를 우선한다

---

## 7. 식사 이름 입력
### 입력 방식
- 자유 텍스트 입력

### 검증
- 빈값 불가

### 의도
기본값은 `아침 / 점심 / 저녁`이지만, 사용자가 아래처럼 변경할 수 있어야 한다.

예:
- `운동 전 식사`
- `오후 간식`
- `늦은 저녁`

주의:
- 완전 자유 입력은 허용하되, 식사 이름을 비워두는 것은 허용하지 않는다

---

## 8. 식사 시간 입력
### 입력 방식
- 각 식사마다 시간 1개 입력
- **HH:MM 형태의 시간 입력 필드** 사용

예:
- `08:00`
- `13:00`
- `19:30`

### 검증
- 빈값 불가
- HH:MM 형식 검증

잘못된 예:
- ``
- `9:3`
- `25:00`
- `12:77`

주의:
- 시작/종료 시간 범위 입력이 아니라, **시간 1개만 입력**한다

---

## 9. 식사 추가
### 버튼 문구
`식사 추가하기`

### 위치
- 식사 목록 바로 아래
- **좌측 정렬**

주의:
- 전체 폭 버튼으로 만들지 않는다
- 페이지 하단 `다음` CTA와 경쟁하지 않게 한다

---

## 10. 식사 삭제
### 방식
- 휴지통 아이콘 사용
- 텍스트 버튼은 사용하지 않는다

### 보조 처리
- 툴팁 또는 접근성 라벨 처리
- 터치 영역은 충분히 크게

예시 접근성 라벨:
- `아침 삭제`
- `점심 삭제`

주의:
- 반복 리스트에서 `삭제` 텍스트 버튼을 여러 개 노출하지 않는다

---

## 11. 식사 순서 변경
### 방식
- 드래그 핸들 사용

### 요구사항
- 드래그 핸들은 명확히 보여야 한다
- 모바일에서도 잡기 쉬운 터치 영역 필요
- 식사 순서를 위/아래로 바꿀 수 있어야 한다

주의:
- 위/아래 화살표 버튼 방식으로 구현하지 않는다

---

## 12. CTA
### 유지 규칙
- 기존 daily-meal 페이지 하단의 `다음` CTA를 그대로 사용한다
- 식사 구조 섹션용 별도 저장 버튼은 추가하지 않는다

즉,
- 목표 설정
- 식사 구조
를 모두 입력한 뒤, 페이지 하단 `다음` 버튼 하나로 다음 단계로 넘어가게 한다.

---

## 13. 디자인 방향
OnFit 디자인 시스템 v1을 참고한다.

### 핵심 방향
- 화이트 기반 미니멀 UI
- 조용한 기본 위에 작은 포인트만 사용
- 식사 구조 섹션은 기존 daily-meal 카드 톤을 해치지 않게 추가
- 과한 장식, 과한 애니메이션 금지

### 구현 포인트
- 목표 설정 영역과 식사 구조 섹션은 충분한 간격 + 얇은 구분선으로 나눈다
- 식사 목록은 한 줄 배치 기준으로 정돈되게 보이게 한다
- 반복되는 식사 블록은 지나치게 무거운 카드형보다, 가볍고 정돈된 입력 행(row) 구조를 우선한다

---

## 14. 금지 사항
아래 변경은 하지 않는다.

- 식사 구조를 별도 페이지로 분리하기
- 식사 구조 섹션에 별도 설명 문구 추가하기
- 식사 이름/시간을 두 줄 구조로 바꾸기
- 식사 시간 입력을 시작/종료 시간 범위로 바꾸기
- `삭제` 텍스트 버튼 반복 노출하기
- 순서 변경을 화살표 버튼 방식으로 바꾸기
- 식사 구조 섹션용 별도 저장 버튼 추가하기
- 기존 daily-meal 목표 설정 영역의 상단 타이틀/설명/모달 규칙을 임의로 수정하기

---

## 15. 구현 우선순위
1. 목표 설정 영역 아래에 식사 구조 섹션 배치
2. 섹션 타이틀 추가
3. 기본 3개 식사 행 추가
4. 식사 이름 + 시간 한 줄 배치 반영
5. 식사 이름 자유 입력 / 빈값 불가 반영
6. 시간 HH:MM 입력 및 검증 반영
7. `식사 추가하기` 버튼 반영
8. 휴지통 아이콘 삭제 기능 반영
9. 드래그 핸들 순서 변경 반영
10. 기존 `다음` CTA와 연결 상태 점검

---

## 16. 최종 구현 기준
구현 결과는 아래 조건을 만족해야 한다.

- 식사 구조 섹션은 daily-meal 기존 카드 안에 있어야 한다
- 목표 설정 영역 아래, `다음` 버튼 위에 배치되어야 한다
- 목표 설정과 식사 구조 사이는 충분한 간격 + 얇은 구분선으로 구분되어야 한다
- 섹션 타이틀은 `하루에 얼마나 드실 수 있으세요?`여야 한다
- 별도 설명 문구는 없어야 한다
- 기본 식사 3개는 `아침 / 점심 / 저녁`이어야 한다
- 식사 수는 최소 1개, 최대 10개여야 한다
- 각 식사는 이름 + 시간을 한 줄에 배치해야 한다
- 비율은 식사 이름 7/10, 시간 3/10이어야 한다
- 식사 이름은 자유 텍스트 입력 가능하되 빈값이면 안 된다
- 식사 시간은 HH:MM 필드여야 하며 빈값 + 형식 검증을 해야 한다
- `식사 추가하기` 버튼은 식사 목록 바로 아래 좌측 정렬이어야 한다
- 삭제는 휴지통 아이콘 + 보조 툴팁/접근성 처리여야 한다
- 순서 변경은 드래그 핸들 방식이어야 한다
- 페이지 하단 CTA는 기존 `다음` 하나를 그대로 사용해야 한다
